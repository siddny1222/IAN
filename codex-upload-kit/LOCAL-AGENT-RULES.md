# 本地 Agent 规则文件怎么用

当前项目已经有一个可直接使用的本地规则文件：

- `/Users/sid/Desktop/new/AGENTS.md`

它的作用是告诉 Codex：

- 这个项目最重要的视觉目标是什么
- 哪些技术约束不能破坏
- 改完代码后该跑哪些检查
- Review 时优先抓什么问题

## 最推荐的用法

把 `AGENTS.md` 放在项目根目录，然后直接让 Codex 在这个仓库里工作。

当前你已经是这种结构了，所以后面无论你在本地 Codex、Codex cloud，还是 GitHub PR review 里跑 Codex，它都能读取这份规则。

## 什么时候会生效

常见场景：

- 在本地 Codex / Codex app 里打开这个项目并开始任务
- 在 Codex cloud 里连接这个仓库后启动任务
- 在 GitHub PR 里用 `@codex review`

## 它是怎么匹配作用域的

Codex 会优先使用离当前文件最近的 `AGENTS.md`。

所以：

- 根目录 `AGENTS.md` 适合放全站规则
- 如果你以后想给某个子目录定更细的规则，也可以在子目录再放一个 `AGENTS.md`

例如：

- 根目录控制全局规则
- `src/pages/AGENTS.md` 控制页面开发规则
- `public/media/AGENTS.md` 控制素材命名和体积规则

## 本地使用步骤

1. 保持根目录有 `AGENTS.md`。
2. 在这个项目目录中打开 Codex。
3. 直接发任务，不需要每次重复讲一遍全部项目规则。
4. 如果你有临时要求，再在这次对话里补充即可。

推荐写法：

```text
在遵守仓库 AGENTS.md 的前提下，优化首页首屏动效，但不要破坏低性能模式。
```

## 如果你想加强规则

你可以继续往 `AGENTS.md` 里补下面这些内容：

- 必跑命令
- 不允许修改的目录
- 页面设计语气
- review 时的严重级别标准
- 素材体积上限

## 这份项目规则最适合放什么

建议放：

- 项目目标
- 视觉与产品边界
- 技术约束
- 验证命令
- review 偏好

不建议放：

- 经常变化的临时任务
- 很长的背景故事
- 与当前仓库无关的个人偏好
